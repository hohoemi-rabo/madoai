# MADOAI（まどあい） — 要件定義書（Part 1: プロジェクト概要）

> **作成日**: 2026-03-20
> **ステータス**: ドラフト v1.0

## 1. プロジェクトの背景と目的

### 1.1 背景

自治体のホームページは、網羅性が求められる性質上、階層が深く、専門用語（お役所言葉）が多い。
目的の情報にたどり着くまでに何度もクリックが必要で、特にスマートフォンからのアクセスでは離脱率が高い。

飯田市ホームページ（https://www.city.iida.lg.jp）も例外ではなく、
「ごみの出し方を知りたいだけなのに、どこを見ればいいか分からない」
「子育て支援の制度が複数の課にまたがっていて全体像がつかめない」
といった課題がある。

一方で、ホームページには市民生活に必要な情報が網羅的に掲載されている。
問題は「情報がない」ことではなく「情報にたどり着けない」ことにある。

### 1.2 目的

自治体ホームページの情報をナレッジベースとして活用し、
市民が自然言語で質問するだけで、必要な情報にたどり着ける対話型AIサービスを提供する。

AIは行政情報の代替ではない。AIは「公式ページへの最短ルート」を提供するナビゲーターである。

すべての回答には元の公式ページへのリンクを必ず含め、
「詳細はこちらの公式ページをご確認ください」と案内するフローを必須とする。

### 1.3 コンセプト

「自治体HP × 対話型AI × マルチテナント」

- 市民は自然言語で行政情報を検索できる
- 自治体HPの既存コンテンツをAIのナレッジベースとして活用する
- 回答には必ず公式ページのURLを提示し、正確性を担保する
- 1つのサービスで複数の自治体（飯田市、高森町、松川町など）を管理できる
- 自治体間の情報比較も可能にし、住民の情報アクセス格差を解消する

### 1.4 ビジネスプランコンテストでの位置づけ

- 個人プロジェクトとして開発し、実績ができたら自治体に提案する
- 飯田市のビジネスプランコンテストへの出展を予定
- 技術力のアピールと、行政DXへの具体的な提案を目的とする

### 1.5 プロジェクト体制

| 役割              | 担当       |
| ----------------- | ---------- |
| エンジニア / 企画 | 開発担当者 |

## 3. スコープ

### 3.1 MVPで実装するもの

- AIチャットページ（Webアプリケーション）
- ナレッジデータの管理機能（管理画面）
- ナレッジデータ用のデータベーステーブル（マルチテナント対応）

- 分析・モニタリング（利用状況 / コスト管理）
- 免責事項・利用規約の表示

### 3.2 MVPで実装しないもの

- ユーザーアカウント / ログイン機能
- LINEボット連携（将来検討）
- 多言語対応（日本語のみ）
- 会話履歴のサーバー保存
- PDF内テキストの自動取得（MVPではPDFリンク案内のみ）
- スクレイピングの自動定期実行（MVPではローカル手動実行）
- ハイブリッド検索（将来拡張）

## 5. AIの人格設計（未確定 — 後日決定）

### 5.1 基本方針

| 項目   | 内容                                                     |
| ------ | -------------------------------------------------------- |
| 名前   | 未定（後日決定）                                         |
| トーン | 優しく親しみやすい口調。事務的すぎず、カジュアルすぎない |
| 一人称 | 未定                                                     |
| 信頼性 | 行政情報を扱うため、正確性と誠実さを最優先               |

### 5.2 回答の制約（確定）

- ナレッジデータにない情報は推測で答えず、「分かりません」と正直に伝える
- 必ず回答の末尾に参照元の飯田市HP公式URLを提示する
- 「詳細はこちらの公式ページをご確認ください」と案内する
- 対象自治体以外の質問には対応範囲外であることを伝える
- 医療・法律・税金に関する質問は「専門家や担当窓口にご相談ください」と案内する

### 5.3 対応範囲外の質問への応答例

**ナレッジにない場合:**
「すみません、その情報はまだ私のデータにないんです。飯田市の公式サイト（https://www.city.iida.lg.jp）で直接お探しいただくか、市役所（0265-22-4511）にお電話でお問い合わせください。」

**対象自治体外の場合:**
「申し訳ありません、現在は飯田市の情報のみ対応しています。他の自治体の情報は順次追加予定ですので、お待ちくださいね。」

### 6.2 Embeddingモデルに関する重要な注意

text-embedding-004 は2026年1月14日に廃止済み。
現在の推奨モデルは `gemini-embedding-001`（テキスト専用、最大3072次元、入力上限2,048トークン）。
pgvector のストレージ効率を考慮し、`output_dimensionality: 768` で運用する。
ドキュメント側は `RETRIEVAL_DOCUMENT`、クエリ側は `QUESTION_ANSWERING` の taskType を指定する。

## 7. データ設計

### 7.1 マルチテナント設計方針

1つのデータベースで複数の自治体を管理する「共有DB・テナントカラム方式」を採用する。
すべての主要テーブルに `municipality_id` カラムを持たせ、自治体ごとにデータを分離する。

将来的に「飯田市と松川町の子育て支援を比較して」のような横断検索にも対応するため、
統合型（1つのチャット画面で複数自治体の情報を返せる構造）で設計する。

### 7.2 新規テーブル: municipalities（自治体マスタ）

| カラム          | 型          | NULL | デフォルト        | 備考                               |
| --------------- | ----------- | ---- | ----------------- | ---------------------------------- |
| id              | uuid        | NO   | gen_random_uuid() | PK                                 |
| code            | text        | NO   | -                 | 自治体コード（例: 202053）、UNIQUE |
| name            | text        | NO   | -                 | 自治体名（例: 飯田市）             |
| name_kana       | text        | YES  | -                 | ふりがな（例: いいだし）           |
| prefecture      | text        | NO   | -                 | 都道府県（例: 長野県）             |
| website_url     | text        | NO   | -                 | 公式HP URL                         |
| scraping_config | jsonb       | YES  | -                 | スクレイピング設定（下記参照）     |
| is_active       | boolean     | NO   | true              | AIの対応対象とするか               |
| created_at      | timestamptz | NO   | now()             |                                    |
| updated_at      | timestamptz | NO   | now()             |                                    |

### 7.9 RPC 関数

**match_pages(query_embedding vector, match_threshold float, match_count int, p_municipality_id uuid DEFAULT NULL) → TABLE**

- コサイン類似度でベクトル検索
- is_active = true のページのみ対象
- p_municipality_id が指定されていれば該当自治体のみ、NULL なら全自治体横断で検索
- 類似度スコア付きで返却（id, municipality_id, source_url, title, category, metadata, content, similarity）

**check_usage_limit(p_session_id text) → jsonb**

- 日次ユーザー制限と月次サイト制限をチェック
- 返却: allowed（真偽値）、daily_remaining（残回数）、reason（制限理由）

**log_ai_usage(p_session_id text, p_municipality_id uuid, p_query text, p_token_input int, p_token_output int, p_matched_pages jsonb, p_response_time_ms int) → void**

- 使用ログを記録
- 日次/月次カウンターをインクリメント

## 8. スクレイピング設計

### 8.1 処理フロー

1.CSVのURLリストから対象カテゴリのURLを絞り込む
2.robots.txt を確認し、クロール可能か検証3.対象ページのHTMLを取得（間隔: 2秒以上）
4.HTMLからテキストを抽出（不要な要素を除去）5.テキストを構造化マークダウンに変換6.メタデータを抽出（タイトル、更新日、担当部署、電話番号など）
7.PDFリンクがあればURLを抽出・記録（本文取得はしない）8.構造化データをJSONファイルとして出力9.管理画面からインポート、または直接DBに投入

### 8.2 実行環境

| 段階 | 環境               | 方法                             |
| ---- | ------------------ | -------------------------------- |
| MVP  | ローカル（Python） | 手動実行スクリプト               |
| 将来 | サーバー or CI/CD  | 定期実行（月次 or 年度替わり時） |

### 8.3 スクレイピング時の配慮事項

- robots.txt を事前に確認し、クロール禁止パスを除外する
- アクセス間隔は最低2秒以上空ける
- User-Agent に連絡先メールアドレスを含める
- 自治体サーバーの負荷が高い時間帯（平日9〜17時）を避ける
- 1回のスクレイピングで取得するページ数を制限する（MVP: 100ページ以下）

### 8.4 飯田市HPのURL構造分析

CSVデータの分析結果:

| URLプレフィックス   | 内容                 | 特徴                                                            |
| ------------------- | -------------------- | --------------------------------------------------------------- |
| `life/2/...`        | カテゴリ別分類ページ | リンク集が多く、本文が薄い（ナビゲーション用）                  |
| `site/gomi/...`     | ごみ特設サイト       | 具体的なコンテンツあり。MVPの主要ターゲット                     |
| `site/kosodate/...` | 子育て応援サイト     | 具体的なコンテンツあり。MVPの主要ターゲット                     |
| `site/yuiturn/...`  | 移住ポータル         | 将来拡張候補                                                    |
| `soshiki/...`       | 組織別ページ         | 部署ごとの詳細情報。子育て関連は soshiki/12/, soshiki/173/ など |
| `life/sub/2/...`    | よくある質問         | FAQ形式。ナレッジ化の価値が高い                                 |

**MVP対象URLの候補:**

ごみ・リサイクル:

- `site/gomi/` 配下全ページ（7ページ）
- `soshiki/19/` 配下の関連ページ（環境課）

子育て:

- `site/kosodate/` 配下全ページ（6ページ）
- `soshiki/12/` 配下（保育家庭課）
- `soshiki/173/` 配下（こども課）
- `soshiki/15/` 配下の予防接種関連（保健課）
- `life/2/6/33/` 配下（子育て支援カテゴリ）
- `life/2/6/35/` 配下（出産・育児カテゴリ）

### 8.5 PDF対応方針

**MVP段階:**

- HTMLテキストのみをナレッジ化する
- PDFリンクが含まれるページには `has_pdf_links: true` フラグを設定
- PDFのURLを `pdf_urls` カラムに記録
- AIの回答時に「詳しくはこちらのPDFをご覧ください」とリンクを案内

**将来拡張:**

- PDFをダウンロードしてテキスト化（OCR or PDFパーサー）
- テキスト化した内容をナレッジデータに統合

### 8.6 コンテンツ抽出ルール

飯田市HPの共通構造に基づき、以下の要素を抽出・除外する。

**抽出対象:**

- `<main>` タグ内のテキスト
- 見出し（h1〜h4）の階層構造
- テーブルデータ（手数料表、日程表など）
- リスト（手続きの手順など）
- ページID、掲載日・更新日
- PDFリンクのURL

**除外対象:**

- ヘッダー・フッターのナビゲーション
- パンくずリスト
- 「Adobe Reader」のダウンロード案内
- サイドバーのメニュー
- 画像（alt テキストは取得）
- アンケートリンク

- 手続きや制度の詳細は「公式ページで最新情報をご確認ください」と案内すること

- 複数の自治体の情報を含む質問には、それぞれの自治体名を明記して回答すること
- 情報が古い可能性がある場合は「〇年〇月時点の情報です」と注記すること

・メタデータ抽出（タイトル、更新日、担当部署、電話番号など）
・PDFリンクの記録

3.構造化データをJSONファイルとして出力4.管理画面から一括インポート5.エンジニアがレビュー・修正
6.AIによるメタデータ自動生成（summary, keywords）→ レビュー
7.Embedding を一括生成
8.AIプロンプトタグを AI で一括生成 → レビュー・調整

### 15.2 情報更新時の運用フロー

1.定期的に（月次 or 年度替わり時）スクレイピングを再実行
2.content_hash を比較し、変更があったページを検出3.管理画面に「要更新」バッジが表示される4.管理画面でナレッジデータを開き、変更点を確認5.必要に応じてナレッジデータを手動で更新・保存
6.Embedding が自動再生成される

### 15.3 新規カテゴリ追加時の運用フロー

1.CSVから新カテゴリのURLを抽出2.スクレイピング → 構造化 → インポート3.レビュー・修正
4.Embedding 生成5.プロンプトタグの追加・再生成

### 15.4 新規自治体追加時の運用フロー

1.管理画面で自治体マスタを追加（名前、コード、HP URL、スクレイピング設定）2.対象ページのURLリストをCSV等で用意3.スクレイピング → 構造化 → インポート（municipality_id を指定）4.レビュー・修正
5.Embedding 生成6.自治体固有のプロンプトタグを生成
7.is_active = true に設定して公開

### 15.5 情報の鮮度に関する注意事項

- ごみの分別ルールや収集日程は年度替わりで変更される可能性がある
- 補助金・助成制度は予算年度ごとに内容が変わる
- ナレッジデータには `scraped_at`（最終スクレイピング日時）を記録し、
  AIの回答時に「この情報は〇年〇月時点のものです」と注記できるようにする
- 年度替わり（4月）前後には重点的にスクレイピングを再実行する

## 19. リスクと対策

| リスク                                         | 影響度 | 対策                                                                |
| ---------------------------------------------- | ------ | ------------------------------------------------------------------- |
| AIが誤った行政情報を回答する                   | 高     | ナレッジベース外の情報は答えない制約 + 必ずソースURL提示 + 免責表示 |
| 飯田市から苦情・停止要請が来る                 | 中     | 免責事項の明示、公式でないことの明記、問い合わせ先の記載            |
| スクレイピングがブロックされる                 | 中     | robots.txt 遵守、適切なクロール間隔、User-Agent明記                 |
| APIコストが想定を超える                        | 中     | 月次上限の設定、ユーザー日次制限、管理画面でのコスト監視            |
| ナレッジデータが古くなる                       | 中     | content_hash による変更検知、年度替わり時の重点更新                 |
| Geminiモデルの廃止・価格変更                   | 低     | 環境変数でモデル名を管理し、差し替え可能な設計                      |
| 悪意のある利用（プロンプトインジェクション等） | 中     | システムプロンプトでの制約、日次制限、ログ監視                      |
